1️⃣ Git を入れる

PC の中で変更履歴を記録するツール本体です。自分の OS のコマンドをコピーして実行します。

Windows（おすすめ: winget）

winget install --id Git.Git -e

macOS（Homebrew）

brew install git

Linux（Debian / Ubuntu 系）

sudo apt update && sudo apt install -y git

winget が使えない場合は git-scm.com のインストーラー（選択肢は既定でOK）。Homebrew が無ければ xcode-select --install でも入ります。確認は新しいターミナルで git --version（command not found の時はターミナルを開き直す）。

2️⃣ コードエディタ（VS Code）を入れる

無料で Git と相性がよく初心者向け。code.visualstudio.com から入れます。Windows は winget でも入ります。

Windows（winget）

winget install --id Microsoft.VisualStudioCode -e

起動して File > Open Folder できればOK。VS Code内ターミナルは Ctrl+` で開けます。

3️⃣ ターミナルを開く

OS使うもの開き方
WindowsPowerShellスタートで「PowerShell」検索
macOSターミナルアプリ > ユーティリティ > ターミナル
Linux端末アプリ一覧から Terminal

Windows は「Git Bash」も可。本教材は PowerShell と macOS/Linux を併記します。

4️⃣ 名前とメールを設定

Commit に記録される名前とメールを、最初に一度だけ設定します（あなたの名前・you@example.com は自分の値に置き換え）。

git config --global user.name "あなたの名前"
git config --global user.email "you@example.com"
git config --global init.defaultBranch main

3行目は既定ブランチ名を main にそろえる設定。メールは GitHub 登録分か、Settings > Emails の noreply アドレス推奨。

設定できたか確認します。

git config --global user.name

5️⃣ GitHub に接続する認証

方法難易度向いている人
A. GitHub CLI（gh）★かんたんこれから始める人（おすすめ）
B. HTTPS + 資格情報マネージャー★かんたんツールを増やしたくない人
C. SSH 鍵★★ふつう複数端末・仕組み理解したい人

A. GitHub CLI（おすすめ） — インストール（自分の OS のもの）。

Windows:

winget install --id GitHub.cli -e

macOS:

brew install gh

ログインして、状態を確認します。

gh auth login
gh auth status

gh auth login の質問は GitHub.com → HTTPS → Yes → ブラウザログイン の順に答えます。Logged in ... と出れば成功。

B. HTTPS: 初回 git push 時にブラウザ認証が出るので、特別な準備は不要です。代替として PAT classic（scope: repo）を発行し、push のパスワード欄に貼る方法もあります。

C. SSH 鍵 — 鍵を作成 → 公開鍵を表示してコピー → 接続テストの順に実行します。

ssh-keygen -t ed25519 -C "you@example.com"
cat ~/.ssh/id_ed25519.pub
ssh -T git@github.com

途中の質問は Enter でOK。表示された公開鍵を GitHub の Settings > SSH and GPG keys に登録します。Hi <ID>! と出れば成功。迷ったら A、困ったら講師へ（認証まわりは環境差が大きいので一人で悩まなくてOK）。

6️⃣ 動作確認

ローカルを使う人は、GitとCommit設定を確認します。

git --version
git config --global user.name

A（GitHub CLI）を選んだ場合:

gh auth status

C（SSH）を選んだ場合:

ssh -T git@github.com

B（HTTPS）は初回push時にブラウザ認証を確認します。実際の clone / push は、自分の練習Repoを作った後に行います。

M1 / Basics

Git と GitHub の基本

GitHub を使った開発で出てくる言葉と機能を、図とセットでイメージできるように整理します。最初に「Git は何をする道具か」「GitHub は何をする場所か」をつかみましょう。

Git と GitHub の違い
名前これは何？役割たとえ
Git自分のPCで動く履歴管理ツールファイルの変更を「いつ・何を変えたか」記録するやり直せるセーブ機能つきのノート
GitHubクラウド上の共有サービスGit の履歴を共有し、相談・レビューするみんなで見られる作業場所

ざっくり言うと Git＝履歴を残す道具、GitHub＝その履歴をチームで共有する場所。この2つはセットで使います。

📖 用語解説

Repository repo
プロジェクトのファイルと変更履歴をまとめて置く場所。1つのアプリや教材ごとに1つ作るのが基本です。本ワークショップでは、Template から参加者ごとに1つ作ります。
main main
チームにとっての「今の正しい状態」を表す基準のブランチ。ここに直接変更せず、Branch と Pull Request を通して反映します。
Issue issue
これからやること・目的・相談を記録する場所。作業を始める前に「何のための変更か」を残しておくと、後から見ても分かりやすくなります。例: Add profile for octocat
Branch branch
main から枝分かれした自分専用の作業場所。本流に影響を与えず安全に変更を試せます。例: add-profile-octocat
Commit commit
変更を履歴に残す保存ポイント。「何を変えたか」を短いメッセージとセットで記録します。例: Add profile for octocat
Pull Request PR
Branch の変更を main に取り込んでよいか相談する場所。差分の確認・コメント・レビュー・履歴がここに集まります。
Review review
Pull Request の変更を他の人が確認してコメントすること。間違い探しだけでなく、意図の確認やチームでの合意形成の場です。
Merge merge
レビューを終えた Pull Request の変更をmain に取り込む操作。これで変更が「チームの正しい状態」に反映されます。
レビューの見方
  • 作業内容が Issue と合っているか
  • 差分に不要な変更が混ざっていないか
  • ファイル名や配置が正しいか
  • 説明が読みやすいか

詳しく見る: GitHub と Git の基本 ／ GitHub の基本機能

M2 / Flow

GitHub での標準的な進め方（GitHub Flow）

GitHub での開発で広く使われている標準的な進め方です。main を安定した状態に保ちながら、小さな変更を Pull Request で取り込みます。特別なルールではなく「みんなが使う基本の型」として身につけましょう。

このワークショップでの運用ルール
  • main に直接変更しない
  • 1つの Issue に対して 1つの Branch を作る
  • Pull Request は小さく作る
  • Review コメントには丁寧に返信する
  • Merge 後は不要になった Branch を削除する

本教材では既定ブランチを main と表記します。画面上で master と表示される場合は読み替えてください。

詳しく見る: GitHub Flow

M3-1 / Repository & Issue

自分の練習Repoを作り、Issueで作業を始める

ここから最後まで、1つの自分専用Repoを使います。Templateで初期ファイルを受け取り、ペアを招待し、MarkdownでバグをIssueにします。

1️⃣ Template からリポジトリを作る

  1. 開く:Tachaan/github-basic-practice
  2. 押す:緑 Use this template → Create a new repository
  3. 入力:Owner は自分、Repository name は github-basic-practice-<github-id>
  4. 選ぶ:Public（ワークショップ推奨）→ Create repository
✅ 作成後: URL が github.com/<github-id>/github-basic-practice-<github-id>、既定Branchが main ならOK。TemplateはForkではなく、初期ファイルと最初の履歴をコピーした独立した新規Repoです。

通常の New repository では README・.gitignore・License を選びます。今回は教材を同じ状態から始めるためTemplateを使い、READMEと.gitignoreをまとめて受け取ります。

📷 ここにスクショ → images/00-use-template.png

2️⃣ Review相手をCollaboratorに追加する

  1. 押す:Settings → Collaborators → Add people
  2. 入力:ペアのGitHub ID（<partner-id>）→ 招待を送る
  3. 相手:通知またはメールから Accept invitation

Public Repoなら招待前でも閲覧・コメントできます。招待で詰まった場合はPRのURLをペアへ共有して先へ進み、MergeはRepo所有者が行います。

3️⃣ バグを確認してIssueを書く

まず Falling Blocks を開き、ブロックが底で止まらないことを確認します。その後、自分のRepoで Issues → New issue → Workshop task を開きます。

M3-2 / Branch & Commit

Repoを開き、Branchで修正してCommitする

自分のRepoをローカルまたはCodespacesで開きます。main は直接変更せず、作業Branchでバグを直してGitHubへ送ります。

4️⃣ Repoを開く

ローカル（基本）

自分のRepoの < > Code → HTTPS でURLをコピーして実行します。

git clone <コピーしたURL>
cd github-basic-practice-<github-id>
code .
Codespaces（代替）

自分のRepoの < > Code → Codespaces → Create codespace on main。開いたクラウドVSCodeのターミナルで、以降の同じgitコマンドを実行します。

📷 ここにスクショ → images/05-code-clone-url.png

5️⃣ アプリを開いて、Branchで1行直す

ローカル: app/falling-blocks/index.html をダブルクリック。Codespaces: 下のコマンドを実行し、Portsに出るURLを開きます。

python -m http.server 8000

バグを確認したら、ターミナルで作業Branchを作ります。

git switch -c fix-falling-blocks-floor-<github-id>
git branch

app/falling-blocks/game.js の FLOOR_ROW を次の1行へ直して保存します。

const FLOOR_ROW = ROWS - 1;
✅ 確認: VSCode左下／git branch が main ではなく作業Branch。再読み込みするとブロックが底で止まります。
📷 ここにスクショ → images/07-vscode-edit.png

6️⃣ 差分を確認してCommit / Pushする

git status
git diff
git add app/falling-blocks/game.js
git commit -m "Fix Falling Blocks floor detection"
git push -u origin fix-falling-blocks-floor-<github-id>
🔑 git diff で関係ない変更がないことを見てからCommitします。Commitは保存ポイント、Pushはその履歴をGitHubへ送る操作です。

M3-3 / Pull Request & Review

Pull Requestで相談し、ペアでReviewする

修正をすぐmainへ入れず、Pull Requestに差分・目的・会話を集めます。ここでIssueとの紐付けとReviewを実際に行います。

7️⃣ Pull Requestを作る

  1. 押す:自分のRepo上部の Compare & pull request
  2. 確認:base: main ← compare: fix-falling-blocks-floor-<github-id>
  3. 入力:PR Templateの Closes #<issue-number> を自分のIssue番号へ置換
  4. 指定:右側 Reviewers でペア（<partner-id>）を選ぶ
  5. 押す:Create pull request → PRのURLをペアへ共有
✅ Files changed は game.js の1行中心。Closes #番号 はIssueとPRを相互リンクし、Merge時にIssueを自動で閉じます。

8️⃣ ペアのPRをReviewする

  1. 開く:ペアから共有されたPRのURL → Files changed
  2. 見る:FLOOR_ROW、不要な差分、Issueとの一致
  3. 書く:行の ＋ からコメント。必要なら Insert a suggestion で修正案を提案
  4. 送る:Review changes → Comment または Approve → Submit review
FLOOR_ROW が ROWS - 1 になっていることを確認しました。
ローカルでもブロックが底で止まりました。

Reviewは責める場ではなく、意図と差分を確認して安心してMergeするための会話です。Repo所有者はコメントへ返信し、修正があれば同じBranchへ再Pushします（PRは自動更新）。

M3-4 / Merge & History

Mergeし、つながった履歴をたどる

Repo所有者がReview済みのPRをMergeします。そこで終わらず、Issue・PR・Commit・1行の由来がつながっていることまで確認します。

9️⃣ Mergeして後片付けする

  1. 押す:Conversation 下部の Merge pull request → Confirm merge
  2. 確認:PRが Merged、対応Issueが Closed
  3. 押す:Delete branch
  4. ローカル:下のコマンドでmainを更新し、作業Branchを削除
git switch main
git pull
git branch -d fix-falling-blocks-floor-<github-id>
📷 ここにスクショ → images/14-merge-pr.png

🔎 10️⃣ 自分の変更を履歴から見つける

git log --oneline --graph -5
git blame app/falling-blocks/game.js
  • git log: Templateの最初のCommitと、自分がMergeした修正が並ぶ
  • git blame: FLOOR_ROW の行に、自分のCommit・名前・日時が出る
  • GitHub検索: is:issue is:closed label:bug / is:pr is:merged
  • PRの Closes #番号、Issueのリンク、Commit SHAを順に開くと「なぜ変えたか」まで戻れる
基本編の完了条件
  • 自分の練習Repo、Issue、Pull Requestがある
  • ペアのReviewが残り、Pull RequestがMergeされている
  • 関連IssueがClosed、作業Branchが削除済み
  • 修正後のアプリが底で止まる
  • log / blame から自分の修正を見つけられる

Advanced / Optional

発展（任意）— コンフリクトとやり直し

基本編の一本の流れを終えた後に扱う、実務でよくあるトラブル対応です。時間に余裕がある場合のみ約20分で紹介します。

コンフリクト: 同じ行を別々に変えた

Gitが「どちらを採用するか決めて」と止まっている状態です。VSCodeで <<<<<<< / ======= / >>>>>>> を消し、正しい内容へ編集してから記録します。

git status
git add <file>
git commit -m "Resolve conflict"
git push

練習: 02. コンフリクト解決

やり直し: まず状態を見て、強い操作を避ける
状況基本の選択
addを戻すgit restore --staged <file>
未Commitの編集を戻すgit restore <file>
push済みを打ち消すgit revert <hash>
履歴を見失ったgit reflog

練習: 03. やり直し・復旧

初心者は単独で使わない: git reset --hard / git push --force / git clean -fd。困ったら新しい操作を重ねず、講師やチームへ相談します。

M4 / Wrap-up

まとめ

Templateから作った1つのRepoに、作業の目的・変更・会話・結果・履歴がつながりました。

ふりかえりの問い
  • Templateから自分のRepoを作ると、何が独立しましたか？
  • IssueのMarkdown・Label・Assignee・@メンションは何を伝えましたか？
  • なぜmainを直接編集せずBranchを作りましたか？
  • Pull RequestとReviewによって何を確認できましたか？
  • Closes #番号 で何がつながりましたか？
  • log / blame から何を調べられましたか？
ゴール: Repository作成 → Issue → Branch → Commit → Pull Request → Review → Merge → 履歴確認を、自分の言葉で説明できること。

練習Repoは学習記録として残しても、Settings → General → Archive しても、削除しても構いません。ペアへのCollaborator権限が不要になったら外します。
