M1 / Basics

Git と GitHub の基本

GitHub Flow で使う言葉と機能を、図とセットでイメージできるように整理します。最初に「Git は何をする道具か」「GitHub は何をする場所か」をつかみましょう。

Git と GitHub の違い
名前これは何？役割たとえ
Git自分のPCで動く履歴管理ツールファイルの変更を「いつ・何を変えたか」記録するやり直せるセーブ機能つきのノート
GitHubクラウド上の共有サービスGit の履歴を共有し、相談・レビューするみんなで見られる作業場所

ざっくり言うと Git＝履歴を残す道具、GitHub＝その履歴をチームで共有する場所。この2つはセットで使います。

📖 用語解説

Repository repo
プロジェクトのファイルと変更履歴をまとめて置く場所。1つのアプリや教材ごとに1つ作るのが基本です。本ワークショップではこのリポジトリが全員の作業場所になります。
main main
チームにとっての「今の正しい状態」を表す基準のブランチ。ここに直接変更せず、Branch と Pull Request を通して反映します。
Issue issue
これからやること・目的・相談を記録する場所。作業を始める前に「何のための変更か」を残しておくと、後から見ても分かりやすくなります。例: Add profile for octocat
Branch branch
main から枝分かれした自分専用の作業場所。本流に影響を与えず安全に変更を試せます。例: add-profile-octocat
Commit commit
変更を履歴に残す保存ポイント。「何を変えたか」を短いメッセージとセットで記録します。例: Add profile for octocat
Pull Request PR
Branch の変更を main に取り込んでよいか相談する場所。差分の確認・コメント・レビュー・履歴がここに集まります。
Review review
Pull Request の変更を他の人が確認してコメントすること。間違い探しだけでなく、意図の確認やチームでの合意形成の場です。
Merge merge
レビューを終えた Pull Request の変更をmain に取り込む操作。これで変更が「チームの正しい状態」に反映されます。

M2 / Flow

GitHub Flow

GitHub Flow は、main を安定した状態に保ちながら、小さな変更を Pull Request で取り込む開発フローです。

キーメッセージ: GitHub Flow は「小さく分けて、見える形で相談し、合意してから取り込む」ための開発の型です。
このワークショップでの運用ルール
  • main に直接変更しない
  • 1つの Issue に対して 1つの Branch を作る
  • Pull Request は小さく作る
  • Review コメントには丁寧に返信する
  • Merge 後は不要になった Branch を削除する

本教材では既定ブランチを main と表記します。画面上で master と表示される場合は読み替えてください。

🛠️ 手順（どこで何をするか）

  1. Issue を作る（GitHub）
    押す:Issues タブ → 右上の緑 New issue → 右下の緑 Submit new issue
    入力:Title と本文（テンプレを貼って <github-id> を置換）
    📷 ここにスクショ → images/03-issue-form-filled.png
  2. clone して VSCode で開く（PC）
    押す:緑 < > Code → HTTPS の URL をコピー
    コマンド:git clone <URL> → cd <repo> → code .
    📷 ここにスクショ → images/05-code-clone-url.png
  3. Branch を作って game.js を修正（PC）
    コマンド:git switch -c fix-falling-blocks-floor-<github-id>
    VSCodeで:const FLOOR_ROW = ROWS - 1;（この1行だけ）→ 保存
    📷 ここにスクショ → images/07-vscode-edit.png
  4. Commit / Push する（PC）
    コマンド:git add app/falling-blocks/game.js → git commit -m "Fix Falling Blocks floor detection"
    送信:git push -u origin fix-falling-blocks-floor-<github-id>
    📷 ここにスクショ → images/08-terminal-commit-push.png
  5. Pull Request を作って Review する（GitHub）
    押す:上部の緑 Compare & pull request → 本文に Closes #<issue-number>
    Review:ペアの PR の Files changed → 右上 Review changes → 緑 Submit review
    📷 ここにスクショ → images/13-review-changes.png
  6. Merge する（GitHub）
    押す:Conversation 最下部の緑 Merge pull request → Confirm merge
    後片付け:git switch main → git pull → git branch -d <branch>
    📷 ここにスクショ → images/14-merge-pr.png
完了条件
  • 自分の Issue がある
  • 自分の Pull Request がある
  • Pull Request に Review コメントが1つ以上ある
  • Pull Request が Merge されている
  • 関連 Issue が Closed になっている、または閉じてよい状態になっている
  • app/falling-blocks/game.js の底判定が修正されている
  • 使い終わった作業 Branch を削除できる状態になっている
作業中の確認ポイント
  • 今いる Branch が main ではなく、自分の作業 Branch になっている（VSCode 左下の Branch 表示／git branch で確認）
  • Pull Request の向きが base: main、compare: fix-falling-blocks-floor-<github-id> になっている
  • Files changed で、app/falling-blocks/game.js の差分が中心になっている
  • Closes #<issue-number> で Issue と PR がつながっている
