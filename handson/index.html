Setup / 環境構築

環境構築 — 開発に必要なツールをそろえる

コードの修正は自分のPCの VSCodeで行います。まず「前提」を確認し、次に Git・VSCode・ターミナル・GitHub認証 をそろえます。何も入っていなくてもOK。順番に用意すれば git clone と git push ができる状態になります。所要 約30〜45分・本編の時間には含みません

前提事項
  • GitHub アカウントを持っていること
    まだ持っていない場合は GitHub アカウントを作成する（公式: GitHubでのアカウントの作成）
  • 開発環境（Git / VSCode / GitHub認証）が用意できていること

🔑 ゴール: 終わると自分のPCから git clone（ダウンロード）と git push（アップロード）ができる状態になります。

1️⃣ Git を入れる

PC の中で変更履歴を記録するツール本体です。自分の OS のコマンドをコピーして実行します。

Windows（おすすめ: winget）

winget install --id Git.Git -e

macOS（Homebrew）

brew install git

Linux（Debian / Ubuntu 系）

sudo apt update && sudo apt install -y git

winget が使えない場合は git-scm.com のインストーラー（選択肢は既定でOK）。Homebrew が無ければ xcode-select --install でも入ります。確認は新しいターミナルで git --version（command not found の時はターミナルを開き直す）。

2️⃣ コードエディタ（VS Code）を入れる

無料で Git と相性がよく初心者向け。code.visualstudio.com から入れます。Windows は winget でも入ります。

Windows（winget）

winget install --id Microsoft.VisualStudioCode -e

起動して File > Open Folder できればOK。VS Code内ターミナルは Ctrl+` で開けます。

3️⃣ ターミナルを開く

OS使うもの開き方
WindowsPowerShellスタートで「PowerShell」検索
macOSターミナルアプリ > ユーティリティ > ターミナル
Linux端末アプリ一覧から Terminal

Windows は「Git Bash」も可。本教材は PowerShell と macOS/Linux を併記します。

4️⃣ 名前とメールを設定

Commit に記録される名前とメールを、最初に一度だけ設定します（あなたの名前・you@example.com は自分の値に置き換え）。

git config --global user.name "あなたの名前"
git config --global user.email "you@example.com"
git config --global init.defaultBranch main

3行目は既定ブランチ名を main にそろえる設定。メールは GitHub 登録分か、Settings > Emails の noreply アドレス推奨。

設定できたか確認します。

git config --global user.name

5️⃣ GitHub に接続する認証

方法難易度向いている人
A. GitHub CLI（gh）★かんたんこれから始める人（おすすめ）
B. HTTPS + 資格情報マネージャー★かんたんツールを増やしたくない人
C. SSH 鍵★★ふつう複数端末・仕組み理解したい人

A. GitHub CLI（おすすめ） — インストール（自分の OS のもの）。

Windows:

winget install --id GitHub.cli -e

macOS:

brew install gh

ログインして、状態を確認します。

gh auth login
gh auth status

gh auth login の質問は GitHub.com → HTTPS → Yes → ブラウザログイン の順に答えます。Logged in ... と出れば成功。

B. HTTPS: 初回 git push 時にブラウザ認証が出るので、特別な準備は不要です。代替として PAT classic（scope: repo）を発行し、push のパスワード欄に貼る方法もあります。

C. SSH 鍵 — 鍵を作成 → 公開鍵を表示してコピー → 接続テストの順に実行します。

ssh-keygen -t ed25519 -C "you@example.com"
cat ~/.ssh/id_ed25519.pub
ssh -T git@github.com

途中の質問は Enter でOK。表示された公開鍵を GitHub の Settings > SSH and GPG keys に登録します。Hi <ID>! と出れば成功。迷ったら A、困ったら講師へ（認証まわりは環境差が大きいので一人で悩まなくてOK）。

6️⃣ 動作確認（clone と push の手前まで）

<owner>/<repo> は講師案内のリポジトリなど、実際の値に置き換えます。

git clone https://github.com/<owner>/<repo>.git
cd <repo>
git log --oneline -5
git fetch

git fetch がエラーなく終われば認証OK（実際の push は本編で行います）。SSH を選んだ場合の clone は git@github.com:<owner>/<repo>.git 形式です。

M1 / Basics

Git と GitHub の基本

GitHub を使った開発で出てくる言葉と機能を、図とセットでイメージできるように整理します。最初に「Git は何をする道具か」「GitHub は何をする場所か」をつかみましょう。

Git と GitHub の違い
名前これは何？役割たとえ
Git自分のPCで動く履歴管理ツールファイルの変更を「いつ・何を変えたか」記録するやり直せるセーブ機能つきのノート
GitHubクラウド上の共有サービスGit の履歴を共有し、相談・レビューするみんなで見られる作業場所

ざっくり言うと Git＝履歴を残す道具、GitHub＝その履歴をチームで共有する場所。この2つはセットで使います。

📖 用語解説

Repository repo
プロジェクトのファイルと変更履歴をまとめて置く場所。1つのアプリや教材ごとに1つ作るのが基本です。本ワークショップではこのリポジトリが全員の作業場所になります。
main main
チームにとっての「今の正しい状態」を表す基準のブランチ。ここに直接変更せず、Branch と Pull Request を通して反映します。
Issue issue
これからやること・目的・相談を記録する場所。作業を始める前に「何のための変更か」を残しておくと、後から見ても分かりやすくなります。例: Add profile for octocat
Branch branch
main から枝分かれした自分専用の作業場所。本流に影響を与えず安全に変更を試せます。例: add-profile-octocat
Commit commit
変更を履歴に残す保存ポイント。「何を変えたか」を短いメッセージとセットで記録します。例: Add profile for octocat
Pull Request PR
Branch の変更を main に取り込んでよいか相談する場所。差分の確認・コメント・レビュー・履歴がここに集まります。
Review review
Pull Request の変更を他の人が確認してコメントすること。間違い探しだけでなく、意図の確認やチームでの合意形成の場です。
Merge merge
レビューを終えた Pull Request の変更をmain に取り込む操作。これで変更が「チームの正しい状態」に反映されます。
レビューの見方
  • 作業内容が Issue と合っているか
  • 差分に不要な変更が混ざっていないか
  • ファイル名や配置が正しいか
  • 説明が読みやすいか

詳しく見る: GitHub と Git の基本 ／ GitHub の基本機能

M2 / Flow

GitHub での標準的な進め方（GitHub Flow）

GitHub での開発で広く使われている標準的な進め方です。main を安定した状態に保ちながら、小さな変更を Pull Request で取り込みます。特別なルールではなく「みんなが使う基本の型」として身につけましょう。

このワークショップでの運用ルール
  • main に直接変更しない
  • 1つの Issue に対して 1つの Branch を作る
  • Pull Request は小さく作る
  • Review コメントには丁寧に返信する
  • Merge 後は不要になった Branch を削除する

本教材では既定ブランチを main と表記します。画面上で master と表示される場合は読み替えてください。

ハンズオン: Falling Blocks の底判定バグを直す

ブロックが底で止まらず落ち続ける原因を app/falling-blocks/game.js で修正します。修正は自分のPCの VSCode、Issue・Pull Request・Review・Merge はGitHub（ブラウザ）で行います。

M4 / Conflicts

コンフリクト解決

同じ行を2人が別々に変えると、Git は自動で決められず コンフリクト（衝突） になります。エラーではなく「どちらを採用するか人に聞いている」サイン。落ち着いて直せば必ず解決できます。

いつ起きる？ 「同じファイルの同じ行」を別々に変えたとき。違う行・違うファイルなら Git がきれいに合体してくれます（コンフリクトになりません）。

🔎 マーカーの読み方

衝突したファイルを開くと、Git が印（マーカー）を書き込みます。

<<<<<<< HEAD
const FLOOR_ROW = ROWS - 1;
=======
const FLOOR_ROW = ROWS;
>>>>>>> main
マーカー意味
<<<<<<< HEADここから下は 自分側 の変更
=======区切り線（上が自分・下が相手）
>>>>>>> mainここまでが 相手側 の変更
🔑 解決とは 3つのマーカーを全部消して、正しい1つの内容に書き直すこと。自分か相手の二択とは限らず、両方を活かした第三の正解でもかまいません。

🛠️ 直す手順（ローカル / VSCode）

  1. どのファイルが衝突したか確認
    コマンド:git status（"Unmerged paths" に出る）
  2. VSCode で開いて書き直す
    押す:マーカー上の Accept Current / Incoming / Both ボタン、または手で編集
    必ず:<<<<<<< ======= >>>>>>> のマーカーを全部消す
  3. 解決を保存して送信
    コマンド:git add <ファイル> → git commit → git push
    確認:git diff --check（残ったマーカーを警告してくれる）
✅ 確認: git status の "Unmerged paths" が消えていれば解決完了。
迷ったとき / 予防のコツ
  • 混乱したら（commit 前なら）git merge --abort で元の状態に戻せる
  • かんたんな衝突は PR 画面の Resolve conflicts → Mark as resolved → Commit merge でもOK
  • 予防: 作業前に git pull ／ PR は小さく短命に ／ こまめに main を取り込む

M5 / Undo & Recovery

やり直し・復旧

Git は履歴を残すので たいていの操作はやり直せます。あわてて強い操作をする前に、まず git status と git log --oneline で状況を確認しましょう。

🧭 「やっちゃった」早見表

状況使うもの
直前の commit メッセージを打ち間違えた（push前）git commit --amend
add したが commit に入れたくないgit restore --staged <file>
編集を全部捨てて元に戻したい（commit前）git restore <file>
main に直接 commit してしまった（push前）新ブランチへ退避 → main を戻す
push 済みの変更を取り消したいgit revert <hash>
履歴を見失ったgit reflog

💡 よくある2つのリカバリ

① main に直接 commit してしまった（push前）
退避:今の変更ごと新しいブランチへ移す 
git switch -c fix-my-work-<github-id>
戻す:main をリモートの状態に合わせる 
git switch main
git reset --hard origin/main
→ 変更は新ブランチに退避され、main は元通り。あとは push → PR の通常フローに戻れます。
② push 済みを取り消す（いちばん安全）
履歴を消さず「打ち消す commit を1つ足す」やり方です。 
git revert <commit-hash>
git push
Web でも Merge 済み PR に Revert ボタンが出れば、打ち消し用の PR を自動で作れます。
⚠️ 触る前に確認（初心者は基本使わない）: git reset --hard（編集や commit を消す）／ git push --force（リモート履歴を上書き）／ git clean -fd（未追跡ファイルを削除）。意味が分かるまで使わないこと。おかしくなったら、新しい操作をする前に講師へ相談してください。
